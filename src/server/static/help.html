ZPR Policy Builder

User Guide — ZPR.org · Applied Invention, LLC

Overview

ZPR Policy Builder is a browser-based tool for authoring, testing, and auditing ZPL (Zero-trust Policy Language) policies. ZPL is defined in ZPR RFC-15.5 and provides a plain-English way to express access control rules: who can access what, under what conditions, and what is explicitly denied.

With ZPR Policy Builder you can:

  • Write ZPL directly in an editor and see parsed classes, entities, and rules instantly
  • Organize policies into a namespace tree with delegated ownership per team or system
  • Define named entity instances using Declare statements, or import them from YAML or LDIF
  • Generate and run policy test suites — including AI-generated adversarial tests using real entity instances
  • Use Policy Studio to generate a complete policy from a plain-English description
  • Use the inline AI Assist chat to author, explain, or refine ZPL
  • Run an AI-powered Policy Audit to detect security risks, shadowing, gaps, and structural issues
  • Save audit results as versioned Audit Reports
  • Upload and browse YAML or LDIF files in the YAML/LDAP Viewer

Getting Started

First Run

When no accounts exist, the application redirects to /setup. Enter a username and password to create the root admin account. This account is automatically granted admin privileges and becomes the owner of the root namespace. Your username becomes the top-level namespace name (e.g. username acme creates root namespace Acme).

Logging In

Use your username and password at the login page. Your session is maintained via a secure cookie. You can also access all API endpoints using a Bearer token (see Profile & API Token).

What You See When You Log In

The active namespace is shown in the Policy Editor header. Two common starting points:

You own your root namespace. You start with a blank policy and can create classes, rules, entities, and sub-namespaces beneath your root.

You were assigned ownership of a sub-namespace. Your namespace tree already contains the namespace you own. Write ZPL within your scope and create further sub-namespaces as needed.

Only ZPL belonging to the active namespace is loaded into the editor. Parent namespaces supply class definitions for cross-namespace references but are not editable from your context.

Staging Input

Writing ZPL

ZPL has four statement types: Define, Allow, Never allow, and Declare.

Defining Classes

Define Employee  as a user    with employee_id, department, optional role.
Define Executive as a user    with role:{CEO, SVP, EVP}.
Define Database  as a service with classification:confidential.
Define AppServer as a server  with classification:corporate, location.

Base classes are user, service, endpoint, and server. Classes inherit parent attributes. The with role:{CEO, SVP, EVP} syntax constrains an attribute to a fixed set of values.

Allow Rules

Allow Employee                      to access Database on AppServer.
Allow Executive with role:{CEO,SVP} to access Database on AppServer.
Allow Partner   with active:true    to access CustomerSystem on SalesServer.

Rules match by class, attribute filter (role:SVP), named entity, tag, or server context (on AppServer).

Never Rules

Never allow Executive with role:EVP to access Database on AppServer.
Never allow FloorWorker to use SystemController.

Never allow is evaluated before any Allow rule — a subject matching a never-allow is denied regardless of any matching allow. There is no override.

Attribute Values Are Exclusive

An entity has exactly one value per attribute. A user with role:EVP cannot simultaneously match role:CEO — they are distinct values. A class defined with role:{CEO, SVP, EVP} means any entity whose role is one of those values qualifies.

Subclasses

Define LegalStaff  as a user with department:legal, optional role.
Define SalesRep    as a user with department:sales, optional region.

An allow rule on a parent class covers all subclasses. An allow on Employee also covers LegalStaff and SalesRep.

Namespaces

Namespaces scope ZPL definitions and rules. Every class and rule belongs to the active namespace. Sub-namespaces let you partition a large policy across teams or systems, each with their own owner.

Click any namespace in the sidebar dropdown to switch context. The active namespace is shown in the Policy Editor header.

To create, rename, delete, or reassign namespaces, use the Namespace Mgnt page. Renaming cascades: all stored ZPL in that namespace and its descendants is re-prefixed automatically.

Cross-namespace class references use dotted notation: Acme.Employee refers to the Employee class in the Acme namespace. The active namespace prefix is stripped in the editor for readability.

Entities

Entities are named instances of ZPL classes — the concrete "who" and "what" of your policy. For example, alice.chen as a LegalStaff member with department:legal, or PatentLibrary as a service with classification:confidential.

Note: In a production ZPR deployment, entity instances typically come from an external directory such as LDAP or Active Directory. The Declare statement is a ZPR Policy Builder extension that lets you describe a small set of named instances directly in the policy for use in simulated rule testing.

Declaring Entities

Click ZPL Declares in the staging toolbar to switch to entity mode, then type:

Declare alice.chen as a LegalStaff with department:legal, role:associate.
Declare james.wu   as an Executive  with role:EVP.
Declare raj.patel  as an Executive  with role:CEO.

Entities are saved as part of namespace ZPL and appear in the Entities tab of the Schema panel.

Importing from YAML or LDIF

Use ↑↓ → Import entities (.ldif) in the toolbar, or the YAML/LDAP menus in the Entities tab. Imported entities are merged into namespace ZPL as Declare statements; duplicates are deduplicated automatically.

YAML format:

- name: alice.chen
  class_name: LegalStaff
  attributes:
    department: legal
    role: associate

LDIF format:

dn: cn=alice.chen,ou=LegalStaff,dc=zpr,dc=policy
cn: alice.chen
objectClass: LegalStaff
department: legal
role: associate

Sample entity files for the Acme demo policy are in Demo_Studio/.

Rule Testing

The Rule Testing page verifies that your ZPL policy behaves as intended. Access it from the left nav, or use the Tests tab in the Schema panel.

Manual Test

In the Manual Test card, select a subject class, enter an action verb, and select an object class. Use +attr under each field to add attribute filters — the key is a dropdown of inherited class attributes; the value is a dropdown when the attribute has predefined values, otherwise free text. Click ▶ Run to evaluate.

The result shows ALLOW or DENY with the matching rule and a full rule trace. Click + Add to suite to save the test.

Generating Tests

Click ⚡ Generate… to open the generator. Choose a count (positive and adversarial), then:

  • From rules — generates class-level test cases derived from each allow rule
  • From entities — uses declared entity instances for realistic tests, plus AI-generated adversarial near-miss cases. Adversarial cases are verified against the engine before inclusion.

Running the Suite

Click ▶ Run All. Each row shows PASS or FAIL. Click a row to expand the payload and full rule trace.

Policy Studio

Policy Studio generates a complete ZPL policy from a plain-English description of your organization and access requirements — no ZPL knowledge required to get started.

Writing a Description

In the Business Description field, describe:

  • Who the users are (roles, departments, levels)
  • What systems or services exist
  • Who should access what, and any explicit exclusions

You can type a description directly, or click ⬆ Upload Description to load one from a .md or .txt file. The Demo_Studio/ folder contains sample description files you can use as a starting point.

Generating the Policy

Click Generate Policy. The AI produces a structured policy with namespaces, classes, and rules. Results appear as expandable namespace blocks, each showing the generated ZPL.

Agent Rationale

After generation, click 💬 Agent Rationale to see the AI's reasoning: an introduction to the policy design, rule-by-rule reasoning for key decisions, and a summary of tradeoffs made.

Deploying the Policy

Click Deploy to Editor to write the generated policy into your namespace tree and open the Policy Editor. The editor runs a ZPL check automatically after deployment so you can see the parsed result immediately.

AI Assist

Click ✨ AI Assistant in the Policy Editor toolbar, or navigate directly. The assistant has the current ZPL as context.

Ask questions, request class or rule suggestions, or ask the assistant to explain existing rules. When the assistant returns a ZPL statement, four action buttons appear:

ButtonAction
AcceptSave the statement directly to the active namespace
StageAdd to the Staged ZPL list for review before saving

The Staged ZPL panel collects statements for batch review. Click ✓ Save All to save them all at once.

Policy Audit

The Policy Audit page uses AI to review your policy for issues that are difficult to spot by reading ZPL directly. The analysis covers the active namespace and all its sub-namespaces.

Fast vs. Deep Mode

  • Fast (default) — results in a few seconds
  • Deep — more thorough; may take up to two minutes

Click 🔍 Run Audit to start.

Reading the Report

A summary appears at the top with issue counts by severity. Each issue card is color-coded:

SeverityMeaning
HIGHSecurity risk or unintended access bypass — review immediately
MEDIUMLikely unintended behavior or unclear policy intent
LOWRedundant rule, cosmetic issue, or minor inconsistency

Each card shows the relevant rule text, a Finding explaining the problem, a Recommendation, and optionally a green ZPL fix block. Categories with no issues are listed at the bottom under No issues in:.

Applying a Suggested Fix

When the analyst can express a fix as a ZPL statement, a green code block with an Accept button appears. Clicking Accept appends the statement to the saved policy in the active namespace. Fixes that require rewriting or deleting an existing rule must be made manually in the Policy Editor.

What the Analyst Checks

  • Shadowing — never-allow rules that block allow rules that look intentional
  • Overly broad — parent-class grants that unintentionally cover confidential or privileged subclasses
  • Class inconsistency — references to undefined classes, unknown attributes, or mismatched values
  • Coverage gaps — classes with no rules (accidental default-deny)
  • Priority conflicts — same-priority overlapping rules with non-deterministic outcomes
  • Redundant rules — rules fully subsumed by another; exact duplicates
  • Never-allow risks — never-allow too broad or replaceable by attribute constraints
  • Structural issues — per-subclass blocking patterns that break when new subclasses are added
The analyst understands ZPL evaluation order: never-allow rules fire first (hard deny, no override), then allow rules are checked. It also understands that attribute values are exclusive per entity — a user with role:EVP cannot simultaneously match role:CEO.

Audit Reports

After a Policy Audit run, click 💾 Save Report to save the result as a versioned audit report. Reports persist across sessions and are listed on the Reports page (accessible from the left nav via the hidden nav target — navigate directly or save from the Audit page).

Reports are titled automatically: Policy Audit — <namespace> — <date> v<n>. Click any report to open it. Click ✕ to delete.

Reports are per-user — you only see reports you saved.

YAML/LDAP Viewer

The YAML/LDAP Viewer (under User Add-ons in the sidebar) lets you upload and browse YAML or LDIF files without importing them into the policy.

Drop a file onto the viewer or click to browse. Accepts .yaml, .yml, and .ldif files.

The viewer renders the content as a formatted document:

  • Entity lists (arrays with class_name and name fields) — grouped by class with one table per class, showing name and all attribute columns
  • Generic object lists — inferred columns from all keys, rendered as a table
  • Dictionaries — section headers per top-level key with nested tables or lists

The sample entity files in Demo_Studio/ (Acme_entities.yaml, Acme_entities.ldif) are designed to be viewed here before importing.

ZPL Config

The ZPL Config page lets you extend the built-in ZPL vocabulary for the active root namespace. Changes apply to all namespaces under that root.

  • Built-in classes — additional base class names that the parser treats as built-ins (not auto-prefixed)
  • Class aliases — map alternate names to canonical class names (e.g. users → user)
  • Custom verbs — additional action verbs beyond the built-in set

System-level entries (from defaults/system_classes.yaml) are shown read-only. Your additions appear separately and can be edited or removed.

Profile & API Token

Click Profile at the bottom of the left nav. From here you can:

  • Update your display name and email
  • Change your password
  • View or regenerate your Bearer token for API access

The Bearer token enables external systems to call the API without a browser session:

Authorization: Bearer <your-token>

Admin

Users with admin privileges can access the admin interface at /admin. The first user (created via /setup) is automatically an admin.

From the admin interface you can:

  • View all users and their root namespaces
  • Create new users with auto-created root namespaces
  • Grant or revoke admin privileges (you cannot revoke your own)
  • Reset any user's password

ZPL Quick Reference

Statement Types

TypeSyntax
Define classDefine <Name> as a <base-class> with <attrs>.
Allow ruleAllow <Subject> to <verb> <Object>.
Never ruleNever allow <Subject> to <verb> <Object>.
Declare entityDeclare <name> as a <Class> with <attrs>.

Base Classes

ClassRole
userHuman actor (subject)
serviceSoftware system (subject or object)
endpointDevice or network location
serverInfrastructure node (used in on clauses)

Rule Evaluation Order

  1. Never-allow rules checked first, highest priority first → match = DENY (hard, no override)
  2. Allow rules checked, highest priority first → match = ALLOW
  3. No match → DENY (default-deny)

Built-in Verbs

access, use, call, read, write, connect to, operate

Release Notes & ZPL Extensions

ZPR Policy Builder implements RFC-15.5 ZPL with the following documented extensions.

Extension 1 — attr:value filters in Allow / Never rules

RFC-15.5 baseline: rule qualifiers support bare tag words only.
This extension: attr:value pairs (and attr:{v1,v2,...} sets) are accepted as qualifiers in Allow and Never rules. Values are matched at rule-evaluation time — equality for single-valued attributes.

Allow Executive with role:{CEO, SVP} to access ITDashboard on AdminServer.
Never allow Executive with role:EVP  to access ITDashboard on AdminServer.

Extension 2 — Cross-namespace dotted class references

RFC-15.5 baseline: class names are single unqualified identifiers.
This extension: fully-qualified dotted names (e.g. Acme.Employee) are valid as class names and parent references. The active namespace prefix is stripped on load and injected on save; cross-namespace references keep their full dotted path.

Allow Acme.Executive with role:{CEO,SVP} to access Admin.ITDashboard on Admin.AdminServer.

Extension 3 — aka aliases and singular/plural class names

RFC-15.5 baseline: class names are exact identifiers.
This extension: class definitions may declare an aka alias. Both forms are accepted interchangeably. Built-in roots: users/user, endpoints/endpoint, services/service, servers/server.

Extension 4 — Declare statement for entity instances

RFC-15.5 baseline: ZPL defines classes and rules only; named instances are not declared in policy text.
This extension: a Declare statement instantiates a named entity of a given class with attribute values. Declared entities are stored with namespace ZPL, appear in the Entities tab, and are used by the test generator.

Declare alice.chen as a LegalStaff with department:legal, role:associate.
Declare james.wu   as an Executive  with role:EVP.
