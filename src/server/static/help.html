ZPR Policy Maker

User Guide — ZPR.org · Applied Invention, LLC

Overview

ZPR Policy Maker is a browser-based editor for writing and testing ZPL (Zero-trust Policy Language) policies. ZPL is defined in ZPR RFC-15.5 and provides a structured, human-readable way to express access control rules across services, users, and endpoints.

With ZPR Policy Maker you can:

  • Write ZPL directly in an editor and see parsed classes and rules instantly
  • Organize policies into a namespace tree with delegated ownership
  • Define named entity instances (e.g. specific users, databases) for realistic testing
  • Generate and run policy test suites — including AI-generated adversarial tests
  • Use AI assistance to help author or explain ZPL
  • Run an AI-powered Rule Analyst to detect shadowing, gaps, and overly broad rules
  • Save analysis results as versioned Audit Reports accessible from the sidebar

Getting Started

First Run

When no accounts exist, the application redirects to /setup. Enter a username and password to create the root admin account. This account is automatically granted admin privileges and becomes the owner of the root namespace.

Logging In

Use your username and password at the login page. Your session is maintained via a secure cookie.

What You See When You Log In

What you see depends on how your account was created. There are two common scenarios:

You own your own root namespace. If an admin created your account to give you an independent space, you own a root namespace (e.g. Acme) that sits at the top of your own hierarchy. You start with a blank policy and are free to create classes, rules, entities, and sub-namespaces beneath it. Nobody else's changes affect your space.

You were delegated ownership of a sub-namespace. If another user created a namespace and assigned you as its owner, you will see that namespace — and any sub-namespaces below it — already present in your namespace selector. There may be ZPL, classes, and rules already defined by the user above you in the hierarchy. Your responsibility is to write ZPL within your assigned namespace and to create any additional sub-namespaces below it that your team needs. You cannot modify ZPL in namespaces above yours.

In either case, the active namespace is shown in the editor toolbar. Only ZPL belonging to that namespace is loaded into the editor — classes and rules from parent or sibling namespaces are separate.

Writing ZPL

ZPL uses three statement types: define, allow, and never allow.

Defining Classes

Define employee as a user with an ID-number, roles
    and optional tags full-time, part-time, and intern.

Define database as a service with content
    and optional tags corporate, employee, financial.

The base classes are user, service, and endpoint. Subclasses inherit attributes and can add their own. The optional tags clause defines allowed tag values for instances.

Allow Rules

Allow full-time employees to access databases.
Allow HR employees to access Payroll-DB.
Allow department:sales employees on managed laptops to access customer databases.

Rules can match by class, attribute value (department:sales), tag, named entity (capitalized), or endpoint context (on managed laptops).

Never Rules

Never allow intern users to access services.
Never allow internet-gateways to access internal services.

never allow takes precedence over any matching allow rule.

Subclasses

Define senior-employee as an employee with clearance-level.
Define internet-gateway as a gateway with external-network-connection:public-internet.

Subclasses inherit all attributes of their parent. Attribute values can be fixed in a subclass definition.

Namespaces

Namespaces scope ZPL definitions and rules. Every class and rule you write belongs to the active namespace. Sub-namespaces allow you to partition a large policy across teams or systems.

To create a sub-namespace, open Namespace Management (Edit button) and click + New under the desired parent. You can specify a different user as the owner — that user can then edit ZPL within that namespace independently.

Renaming a namespace cascades: all stored ZPL in that namespace and its descendants is re-prefixed automatically.

Entities

Entities are named instances of ZPL classes — concrete subjects and objects that exist in your actual environment. For example, Alice as an employee with department: engineering, or Payroll-DB as a database with tags: financial.

Creating Entities

In the Entities tab, click + New entity. Enter the class name, a display name, and any attributes as key/value pairs.

Importing and Exporting

Use the YAML or LDIF menus to import or export entities. The YAML format:

entities:
  - class_name: employee
    name: Alice
    attributes:
      department: engineering
      roles: developer
      tags: full-time
  - class_name: database
    name: Payroll-DB
    attributes:
      tags: financial, employee

LDIF format uses standard LDAP attribute notation.

Using Entities in Tests

Once entities are defined, the Generate using Entities button in the Tests tab creates a test suite using real entity names and attributes, including AI-generated adversarial tests that should be denied.

Tests

The Tests tab lets you verify that your ZPL policy behaves as expected.

Manual Test

Enter a subject class, action, and object class (and optionally attributes) and click Run. The result shows allow or deny with the matching rule.

Click + Add to save the current test to the test suite.

Generating Tests

Click Generate… to open the generator modal. Two options:

  • Generate from rules — creates one allow test and one deny test per rule, using class-level payloads
  • Generate using Entities — uses your defined entity instances for realistic tests, plus AI-generated adversarial near-miss tests

Specify how many adversarial tests to generate. The system verifies each adversarial test against the engine and discards any that do not actually result in a deny.

Running Tests

Check the tests you want to run and click Run. Each test shows pass or fail with the engine result. Tests expecting allow pass if the engine allows; tests expecting deny pass if the engine denies.

AI Assist

Click ⊕ Assist in the editor toolbar to open the AI assistant panel. You can ask questions about ZPL, request help authoring rules, or ask the assistant to explain what existing rules do.

The assistant has access to the current ZPL editor content as context. Suggested edits can be applied directly to the editor.

Rule Analyst

The Rule Analyst uses AI to review your policy and identify issues that are difficult to spot by reading ZPL directly — things like rules that cancel each other out, access grants that are broader than intended due to class inheritance, or protective never-allow rules that should exist but don't.

Click 🔍 Rule Analyst in the ZPL editor toolbar to run an analysis of the active namespace and all its sub-namespaces. The analysis typically completes in a few seconds.

Reading the Report

Results appear in a modal with a summary at the top and a list of issues below. Each issue is color-coded by severity:

SeverityMeaning
HIGHSecurity risk or unintended access bypass — review immediately
MEDIUMLikely unintended behavior or unclear policy intent
LOWRedundant rule, cosmetic issue, or minor inconsistency

Each issue card shows the relevant rule text, a one-sentence Finding explaining the problem, and a one-sentence Fix recommendation. Issue cards are collapsed by default — click a card to expand it.

Categories with no issues found are listed at the bottom under No issues in:.

Applying a Suggested Fix

When the analyst can express the fix as a concrete ZPL statement (typically a new never-allow or allow rule), a green code block with a + Add button appears inside the issue card. Clicking + Add appends the suggested statement to the ZPL editor and immediately re-parses the policy. You can then review the change and click Save to persist it.

Not all issues have a + Add button — fixes that require rewriting or deleting an existing rule must be made manually.

Fast vs. Deep Mode

By default the analyst uses a fast AI model and returns results in a few seconds. For a more thorough analysis — covering subtler issues and more edge cases — click the Deep ▶ button in the modal header. Deep mode uses a more capable model and may take up to two minutes.

The analyst examines the combined ZPL across the active namespace and all its sub-namespaces. Switch to the namespace you want to analyze before opening the analyst.

What the Analyst Checks

  • Shadowing — never-allow rules that block allow rules that look intentional; unreachable rules
  • Overly broad rules — parent-class grants that unintentionally cover confidential or privileged subclasses
  • Class inconsistencies — references to undefined classes, unknown attribute names, or mismatched values
  • Coverage gaps — classes with no rules (accidental default-deny); missing never-allow for sensitive verbs
  • Priority conflicts — rules at the same priority with overlapping subjects and objects, giving non-deterministic results
  • Redundant rules — rules fully subsumed by another rule; exact duplicates
  • Never-allow risks — guards that are too broad; missing hard blocks on destructive verbs for sensitive classes
  • Structural issues — per-subclass blocking patterns that break when new subclasses are added; namespace scatter

Audit Reports

After a Rule Analyst run completes, you can save the result as a versioned Audit Report. Reports are stored in the database and persist across sessions.

Saving a Report

After the analyst returns results, a 💾 Save button appears in the modal header. Click it to save. The report is titled automatically:

Policy Audit — <namespace> — <date> v<n>

The version number increments for each report saved for the same namespace — so running the analyst weekly on Corp.WS produces Policy Audit — WS — 2026-05-03 v1, v2, and so on.

Viewing Saved Reports

Saved reports appear in the Reports section at the bottom of the left sidebar. Click the ▶ toggle to expand it. Reports are listed newest-first with the namespace name, date, and a deep badge if the report came from a deep-mode analysis.

Click any report in the list to open it in the analyst modal for review.

Deleting a Report

Each report row has an ✕ button on the right. Click it to permanently delete that report. You will be asked to confirm.

Reports are per-user — you only see reports you saved. The report stores the structured analysis result as JSON; the visual formatting is applied each time you open it, so it always renders in the current style.

Profile & API Token

Click Profile in the header to open the profile panel. From here you can:

  • Update your display name and email
  • Change your password
  • View or regenerate your Bearer token for API access

The Bearer token can be used by external systems to call the API without a browser session:

Authorization: Bearer <your-token>

Admin

Users with admin privileges can access the admin interface at /admin. The first user created (via /setup) is automatically an admin.

From the admin interface you can:

  • View all users and their root namespaces
  • Create new users with auto-created root namespaces
  • Grant or revoke admin privileges (checkbox per user)
  • Reset any user's password
  • Delete users (blocked if the user owns a root namespace — delete the namespace first)
  • View the full namespace tree with owners

Release Notes & ZPL Extensions

ZPR Policy Builder v3 implements RFC-15.5 ZPL with the following documented extensions.

Extension 1 — attr:value filters in Allow / Never rules

RFC-15.5 baseline: rule qualifiers support bare tag words only.
This extension: attr:value pairs are also accepted as qualifiers before the class name in Allow and Never rules. The value is matched against the class's attribute at rule-evaluation time — equality for single-valued attributes, containment for multi-valued ones.

Never allow backup:nightly servers to access backup-services.
Allow role:engineer employees to access code-repos.
Never allow clearance:low users to access classified services.

Extension 2 — Cross-namespace dotted class references

RFC-15.5 baseline: class names are single unqualified identifiers.
This extension: fully-qualified dotted names (e.g. corp.employee) are valid as class names and parent references in both Define and Allow/Never statements. The active namespace prefix is stripped on load and injected on save; cross-namespace references keep their full dotted path in the editor.

Define employee as a corp.employee with department:hr.
Allow corp.contractors to access corp.hr.employee-db.

Extension 3 — aka aliases and singular/plural class names

RFC-15.5 baseline: class names are exact identifiers.
This extension: class definitions may declare an aka alias (typically a singular or plural variant). Both forms are accepted interchangeably throughout ZPL statements. Built-in roots: users/user, endpoints/endpoint, services/service, servers/server.

Define employee aka employees as a user.
Allow employees to access gitlab.   # both forms valid
