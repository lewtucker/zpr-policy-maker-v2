"Policy Syntax\nA Zero Trust Packet Routing (ZPR) policy consists of one or more policy statements. A policy statement is an expression of intent written in a specific syntax.\nZPR policy syntax differs depending on the location of the source and target VCN, but the security attribute usage is the same regardless. See the following sections for more details.\n\nSecurity Attributes in ZPR Policy🔗\nSecurity attributes identify the VCN and endpoints within the VCN. \nA security attribute is made up of a security attribute namespace and a security attribute key separated with a period, and a value separated with a colon:\n\nFor example, for in applications.app:fin-network the values are as follows: \n\napplications is the security attribute namespace\n\napp is the security attribute key\n\nfin-network is the value\n\nSecurity attribute namespaces, and security attribute keys and values are constrained by specific limits. Importantly, security attribute namespaces and security attribute keys must not contain a space or a period character. Values, however, can contain spaces, periods, and single quotation marks. If a security attribute whose value contains more than the allowed number of characters is referenced, the whole security attribute clause is enclosed within single quotation marks. Any single quotation mark character in the value must be escaped with another single quotation mark character. For example:\n\napp:fin-network\n\noracle-zpr.app:fe-nodes\n\nmy-corp.biz:hr\n\n'my-corp.biz:dev and test db'\n\n Note If the namespace of a security attribute is omitted, ZPR defaults to the oracle-zpr namespace.\nSee Security Attributes for more information about security attributes and the permissions required to use them.\n\nPolicy Syntax for Endpoints in the Same VCN🔗\nZPR policy statements use the following syntax and rules when the source and target reside in the same VCN:\nin <location> <command> <source endpoint> <verb> <destination endpoint>\n\n<location> is required, and it must be in the form in <security attribute> VCN. <security attribute> must be specified, and it can be only a single security attribute.\n\n<command> must be allow. \n\n<source endpoint> must be security attribute, ip address, all-endpoints, or osn-services-ip-addresses. \n\n<verb> must be to connect to.\n\n<destination endpoint> must be security attribute, ip address, all-endpoints, or osn-services-ip-addresses. \n\n Note\nosn-services-ip-addresses refers to all addressable IPv4 and IPv6 IP ranges used by OCI services.\nFor example, the following policy statement expresses intent to allow traffic between endpoints within the same VCN:\nin app:fin-network VCN allow app:web endpoints to connect to app:store endpoints\nThe VCN is identified by its security attribute and is subject to the policy referencing it. The allow statement applies to each VCN with that security attribute. The endpoint clauses identify either the source or the target of traffic with the specified security attribute within a VCN:\n\n The all-endpoints keyword signifies any endpoint inside or outside of the VCN regardless of whether it carries any security attributes. The source endpoint and target endpoint can't both be all-endpoint. One must be identified (endpoint attribute list).\nTraffic between endpoints can be further limited in a policy by filtering on the ip-address and one or more of the allowed network filter attributes: protocol, protocol.icmp.type, protocol.icmp.code, and connection-state.\n\nip-address or osn-services-ip-addresses can be a target or a source. However, you can't use ip-address and osn-services-ip-addresses on both the source and target endpoints; ip-address and osn-services-ip-addresses must be either the source or the target. For example:\nin applications.apps:app1 VCN allow '10.0.0.0/16' to connect to apps:app1 endpoints\n\nPolicy Syntax for Endpoints in Two VCNs🔗\nZPR policy statements use the following syntax and rules when the source and target VCNs reside in the same region and tenancy, and both VCNs use security attributes:\n<command> <source endpoints> in <source location> to <verb> <destination endpoints> in <destination location>\n\n<command> must be allow. \n\n<source endpoint> must be security attribute. Note To refer to an endpoint by IP address or CIDR range, use the policy syntax for a single VCN.\n\n<source location> is required, and it must be in the form in <security attribute> VCN.\n<security attribute> must be specified, and it can be only a single security attribute.\n\n<verb> must be to connect to.\n\n<destination endpoint> must be security attribute.\n\n<destination location> is required, and it must be in the form in <security attribute> VCN.\n<security attribute> must be specified, and it can be only a single security attribute.\n\nFor example, the following policy statement expresses intent to allow traffic from or to endpoints in two VCNs within the same region:\nallow applications.app:webserver endpoints in applications.vcn:A VCN to connect to database.database:MySQL endpoints in database.vcn:B VCN\nThe VCNs are identified by their security attributes and are subject to the policies referencing them. The allow statement applies to each VCN with those security attributes. The endpoint clauses identify either the source or the target of traffic with the specified security attribute within a VCN:\n\nTraffic to and from endpoints can be further limited in policy by filtering with one or more of the allowed network filter attributes: protocol, protocol.icmp.type, protocol.icmp.code, and connection-state.\n\nPolicy Syntax for Other Scenarios🔗\nTo allow traffic to or from a source or destination without a defined security attribute (such as another VCN, a different region, an on-premises network, or the internet), specify the IP address or CIDR block in a ZPR policy using the single VCN policy syntax.\nFor example, the following policy statement expresses intent to allow traffic from or to endpoints at the specified IP address regardless of where there resource resides or whether it has security attributes applied to it: \nin front-end:network VCN allow loadbalancer:web to connect to '0.0.0.0/0'"
