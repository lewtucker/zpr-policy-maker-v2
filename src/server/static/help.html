ZPR Policy Builder

User Guide — ZPR.org · Applied Invention, LLC

Overview

ZPR Policy Builder is a browser-based editor for writing and testing ZPL (Zero-trust Policy Language) policies. ZPL is defined in ZPR RFC-15.5 and provides a structured, human-readable way to express access control rules across services, users, and endpoints.

With ZPR Policy Builder you can:

  • Write ZPL directly in an editor and see parsed classes, entities, and rules instantly
  • Organize policies into a namespace tree with delegated ownership
  • Define named entity instances (e.g. specific users, databases) using Declare statements
  • Generate and run policy test suites — including AI-generated adversarial tests
  • Use AI assistance to help author or explain ZPL
  • Run an AI-powered Policy Audit to detect shadowing, gaps, and overly broad rules
  • Save analysis results as versioned Audit Reports accessible from the Reports page

Getting Started

First Run

When no accounts exist, the application redirects to /setup. Enter a username and password to create the root admin account. This account is automatically granted admin privileges and becomes the owner of the root namespace.

Logging In

Use your username and password at the login page. Your session is maintained via a secure cookie.

What You See When You Log In

What you see depends on how your account was created. There are two common scenarios:

You own your own root namespace. If an admin created your account to give you an independent space, you own a root namespace (e.g. Acme) that sits at the top of your own hierarchy. You start with a blank policy and are free to create classes, rules, entities, and sub-namespaces beneath it.

You were delegated ownership of a sub-namespace. If another user created a namespace and assigned you as its owner, you will see that namespace — and any sub-namespaces below it — already present in your namespace tree. Your responsibility is to write ZPL within your assigned namespace and to create any additional sub-namespaces below it that your team needs.

The active namespace is shown in the Policy Editor header. Only ZPL belonging to that namespace is loaded into the editor — classes and rules from parent or sibling namespaces are separate.

Layout & Navigation

The application has a left navigation sidebar and a main content area. The sidebar contains:

  • Namespace tree — click any namespace to switch the active context; use + to create a sub-namespace
  • Pages — links to each section of the application
  • Profile / Sign out — at the bottom of the sidebar

The Policy Editor is the main workspace. It is split vertically into two zones: the staging input at the top and the Definitions panel below. A draggable divider separates them.

Writing ZPL

ZPL uses four statement types: Define, Allow, Never allow, and Declare*.

* Declare is a ZPR Policy Builder extension — see the Entities section below.

Defining Classes

Define employee as a user with an ID-number, roles
    and optional tags full-time, part-time, and intern.

Define database as a service with content
    and optional tags corporate, employee, financial.

The base classes are user, service, and endpoint. Subclasses inherit attributes and can add their own. The optional tags clause defines allowed tag values for instances.

Allow Rules

Allow full-time employees to access databases.
Allow HR employees to access Payroll-DB.
Allow department:sales employees on managed laptops to access customer databases.

Rules can match by class, attribute value (department:sales), tag, named entity, or endpoint context (on managed laptops).

Never Rules

Never allow intern users to access services.
Never allow internet-gateways to access internal services.

Never allow takes precedence over any matching Allow rule.

Subclasses

Define senior-employee as an employee with clearance-level.
Define internet-gateway as a gateway with external-network-connection:public-internet.

Subclasses inherit all attributes of their parent. Attribute values can be fixed in a subclass definition.

Namespaces

Namespaces scope ZPL definitions and rules. Every class and rule you write belongs to the active namespace. Sub-namespaces allow you to partition a large policy across teams or systems.

Click any namespace in the sidebar tree to switch to it. The active namespace is highlighted and shown in the Policy Editor header.

To create, rename, delete, or reassign namespaces, use the Namespace Manager page (accessible from the left nav). You can also create a sub-namespace quickly by clicking + next to the namespace section header in the sidebar.

Renaming a namespace cascades: all stored ZPL in that namespace and its descendants is re-prefixed automatically.

Entities

Entities are named instances of ZPL classes — concrete subjects and objects that exist in your actual environment. For example, Alice as a warehouse-manager with department: operations, or Payroll-DB as a database with tags: financial.

Note: In a production ZPR deployment, entity instances are typically defined in an external directory such as LDAP or Active Directory — not in ZPL policy text. The Declare statement is a ZPR Policy Builder extension added specifically to facilitate simulated rule testing. It lets you describe a small set of named instances directly in the policy editor so that the test generator can produce realistic, entity-specific test cases without requiring a live directory integration.

Creating Entities

Click the Entities tab in the staging input zone. The textarea switches to entity mode — type one or more Declare statements and click Save:

Declare Alice as a warehouse-manager with department: operations, shift: day.
Declare Payroll-DB as a database with tags: financial, employee.

Declared entities are saved alongside class and rule definitions in the namespace ZPL and appear in the Entities tab of the Definitions panel.

Importing and Exporting

Entities can also be imported from YAML or LDIF files. In the Definitions panel Table view, the Entities tab has import/export controls. The YAML format:

entities:
  - class_name: employee
    name: Alice
    attributes:
      department: engineering
      roles: developer
      tags: full-time
  - class_name: database
    name: Payroll-DB
    attributes:
      tags: financial, employee

LDIF format uses standard LDAP attribute notation. Imported entities are merged into the namespace ZPL as Declare statements; duplicates are deduplicated automatically.

Using Entities in Tests

Once entities are defined, the Generate using Entities option in the Rule Testing generator creates a test suite using real entity names and attributes, including AI-generated adversarial tests.

Rule Testing

The Rule Testing page lets you verify that your ZPL policy behaves as expected. Access it from the left nav.

Manual Test

Enter a subject class, action, and object class (and optionally per-field attributes) and click ▶ Run. The result shows ALLOW or DENY with the matching rule and a full rule trace. Click + Add to save the current test to the test suite.

Generating Tests

Click ⚡ Generate… to open the generator. Two options:

  • Generate from rules — creates positive and adversarial tests using class-level payloads derived from each allow rule
  • Generate using Entities — uses your declared entity instances for realistic tests, plus AI-generated adversarial near-miss cases

Specify how many positive and adversarial tests to generate. The system verifies each adversarial test against the engine before including it — tests that are not actually denied are discarded.

Running Tests

Click ▶ Run All to run the full suite. Each row shows PASS or FAIL. Click a row to expand the payload detail and rule trace.

AI Assist

Click ✨ AI Assist in the Policy Editor staging toolbar, or navigate directly via the left nav. The assistant has access to the current ZPL as context.

You can ask questions about ZPL, request help authoring rules, or ask the assistant to explain what existing rules do. When the assistant returns a ZPL statement, four action buttons appear:

ButtonAction
AcceptImmediately save the statement to the active namespace policy
StageMove to the Staged ZPL list for review before saving

EditLoad into the chat input for modification before sending

The Staged ZPL panel collects statements you want to review together. Click ✓ Save All to save the full staged set to the policy at once, or use the per-item Edit / Delete buttons to refine first.

Click ✓ Done — Back to Editor when finished.

Policy Audit

The Policy Audit page uses AI to review your policy and identify issues that are difficult to spot by reading ZPL directly — things like rules that cancel each other out, access grants that are broader than intended due to class inheritance, or missing protective Never allow guards.

Navigate to Policy Audit in the left nav. The analysis covers the active namespace and all its sub-namespaces.

Fast vs. Deep Mode

Select the analysis depth before running:

  • Fast (default) — quick analysis, results in a few seconds
  • Deep — more thorough analysis covering subtler issues and edge cases; may take up to two minutes

Click 🔍 Run Audit to start.

Reading the Report

Results appear with a summary at the top and a list of issues below. Each issue is color-coded by severity:

SeverityMeaning
HIGHSecurity risk or unintended access bypass — review immediately
MEDIUMLikely unintended behavior or unclear policy intent
LOWRedundant rule, cosmetic issue, or minor inconsistency

Each issue card shows the relevant rule text, a Finding explaining the problem, and a Fix recommendation. Click a card to expand it. Categories with no issues are listed at the bottom under No issues in:.

Applying a Suggested Fix

When the analyst can express the fix as a ZPL statement, a green code block with a + Add button appears inside the card. Clicking + Add appends the statement to the Policy Editor and immediately re-parses. Review and click Save to persist the change.

Fixes that require rewriting or deleting an existing rule must be made manually in the Policy Editor.

What the Analyst Checks

  • Shadowing — Never allow rules that block Allow rules that look intentional; unreachable rules
  • Overly broad rules — parent-class grants that unintentionally cover confidential or privileged subclasses
  • Class inconsistencies — references to undefined classes, unknown attribute names, or mismatched values
  • Coverage gaps — classes with no rules (accidental default-deny); missing Never allow for sensitive verbs
  • Priority conflicts — rules at the same priority with overlapping subjects and objects
  • Redundant rules — rules fully subsumed by another rule; exact duplicates
  • Never-allow risks — guards that are too broad; missing hard blocks on destructive verbs for sensitive classes
  • Structural issues — per-subclass blocking patterns that break when new subclasses are added; namespace scatter

Reports

After a Policy Audit run completes, click 💾 Save in the audit page header to save the result as a versioned Audit Report. Reports are stored in the database and persist across sessions.

Reports are titled automatically:

Policy Audit — <namespace> — <date> v<n>

The version number increments for each report saved for the same namespace on the same date.

View saved reports by navigating to Reports in the left nav. Reports are listed newest-first with the namespace name, date, and a deep badge if the report came from a deep-mode analysis. Click any report to open it. Click the ✕ button on a report row to permanently delete it.

Reports are per-user — you only see reports you saved.

Profile & API Token

Click Profile at the bottom of the left nav to open the profile panel. From here you can:

  • Update your display name and email
  • Change your password
  • View or regenerate your Bearer token for API access

The Bearer token can be used by external systems to call the API without a browser session:

Authorization: Bearer <your-token>

Admin

Users with admin privileges can access the admin interface at /admin. The first user created (via /setup) is automatically an admin.

From the admin interface you can:

  • View all users and their root namespaces
  • Create new users with auto-created root namespaces

Release Notes & ZPL Extensions

ZPR Policy Builder v3 implements RFC-15.5 ZPL with the following documented extensions.

Extension 1 — attr:value filters in Allow / Never rules

RFC-15.5 baseline: rule qualifiers support bare tag words only.
This extension: attr:value pairs are also accepted as qualifiers before the class name in Allow and Never rules. The value is matched against the class's attribute at rule-evaluation time — equality for single-valued attributes, containment for multi-valued ones.

Never allow backup:nightly servers to access backup-services.
Allow role:engineer employees to access code-repos.
Never allow clearance:low users to access classified services.

Extension 2 — Cross-namespace dotted class references

RFC-15.5 baseline: class names are single unqualified identifiers.
This extension: fully-qualified dotted names (e.g. corp.employee) are valid as class names and parent references in both Define and Allow/Never statements. The active namespace prefix is stripped on load and injected on save; cross-namespace references keep their full dotted path in the editor.

Define employee as a corp.employee with department:hr.
Allow corp.contractors to access corp.hr.employee-db.

Extension 3 — aka aliases and singular/plural class names

RFC-15.5 baseline: class names are exact identifiers.
This extension: class definitions may declare an aka alias (typically a singular or plural variant). Both forms are accepted interchangeably throughout ZPL statements. Built-in roots: users/user, endpoints/endpoint, services/service, servers/server.

Define employee aka employees as a user.
Allow employees to access gitlab.   # both forms valid

Extension 4 — Declare statement for entity instances

RFC-15.5 baseline: ZPL defines classes and rules only; named instances are referenced in rules by convention but are not declared in policy text.
This extension: a Declare statement explicitly instantiates a named entity of a given class, with optional attribute values. Declared entities are stored in ZPL alongside class definitions and rules, appear in the Entities tab of the Definitions panel, and are used by the test generator.

Declare Alice as a warehouse-manager with department: operations, shift: day.
Declare Packer-1 as a packing-machine with zone: packing, status: active.

# Named entities can then be referenced directly in rules:
Allow Alice to access hr-database.
Never allow Ted to read payroll-system.

Use the Entities tab in the Policy Editor staging zone to switch the input hint to entity declaration syntax. YAML and LDIF entity files can still be imported via the Entities tab in the Definitions panel; imported entities are automatically converted to Declare statements and merged into the namespace ZPL — duplicates are deduplicated on save.
